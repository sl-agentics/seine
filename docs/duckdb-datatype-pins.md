## D. WHERE semantics: only TRUE passes (UNKNOWN filtered)

- `t_where(v) = [1, NULL, 5]`; `WHERE v > 2` -> rows: [5]
- `WHERE NOT (v > 2)` -> rows: [1] (NULL row excluded from BOTH — 3VL, not complement)

## F. Join semantics: NULL keys never equi-join

- `jl JOIN jr ON jl.k = jr.k` -> [('l1', 'r1')] (the NULL-NULL pair does NOT join)
- `... ON jl.k IS NOT DISTINCT FROM jr.k` -> [('l1', 'r1'), ('lN', 'rN')] (null-safe join for reference)

## H. Grouping / DISTINCT: NULLs are ONE group (TMS-key relevance)

- `GROUP BY v` over [1, NULL, 3] -> [(1, 1), (3, 1), (None, 1)]
- `DISTINCT` over [NULL, NULL, 1] -> [1, None] (nulls collapse)

## I. DOUBLE NaN (the ingest-normalization rationale)

| expression | result |
|---|---|
| `'NaN'::DOUBLE = 'NaN'::DOUBLE` | True |
| `'NaN'::DOUBLE > 1e308` | True |
| `'NaN'::DOUBLE = NULL` | NULL |
| `isnan('NaN'::DOUBLE)` | True |

DuckDB treats NaN as a VALUE (equal to itself, sorts greatest) — one
more reason pandas-NaN normalizes to NULL at the Seine boundary for
nullable float fields rather than leaking IEEE semantics into 3VL.

## J. DECIMAL semantics

| expression | result | note |
|---|---|---|
| `typeof(1.23)` | DECIMAL(3,2) | SQL decimal literal type |
| `typeof(1.23::DECIMAL(18,3))` | DECIMAL(18,3) | explicit cast |
| `1.10::DECIMAL(3,2) = 1.1::DECIMAL(2,1)` | True | cross-scale value equality |
| `1.10::DECIMAL(3,2) = 1` | False | decimal vs integer |
| `0.1::DECIMAL(2,1) + 0.2::DECIMAL(2,1)` | 0.3 | exact addition (no 0.30000000000000004) |
| `typeof(0.1::DECIMAL(2,1) + 0.2::DECIMAL(2,1))` | DECIMAL(3,1) | addition result type |
| `typeof(1.5::DECIMAL(2,1) * 2.5::DECIMAL(2,1))` | DECIMAL(4,2) | multiplication result type |
| `1.005::DECIMAL(18,3)::DECIMAL(18,2)` | 1.01 | scale-reduction rounding |
| `1.015::DECIMAL(18,3)::DECIMAL(18,2)` | 1.02 | scale-reduction rounding (odd) |
| `(-1.005)::DECIMAL(18,3)::DECIMAL(18,2)` | -1.01 | negative rounding |
| `0.1::DECIMAL(2,1) = 0.1::DOUBLE` | True | decimal vs double compare |
| `typeof(sum(x)) FROM (VALUES (1.25::DECIMAL(10,2))) t(x)` | DECIMAL(38,2) | SUM result type |
| `typeof(avg(x)) FROM (VALUES (1.25::DECIMAL(10,2))) t(x)` | DOUBLE | AVG result type |
| `typeof(min(x)) FROM (VALUES (1.25::DECIMAL(10,2))) t(x)` | DECIMAL(10,2) | MIN result type |
| `sum(x) FROM (VALUES (0.1::DECIMAL(2,1)),(0.2::DECIMAL(2,1)),(0.3::DECIMAL(2,1))) t(x)` | 0.6 | exact SUM |
| `avg(x) FROM (VALUES (1::DECIMAL(2,0)),(2::DECIMAL(2,0))) t(x)` | 1.5 | AVG value |

Overflow behavior:
- `99.99::DECIMAL(4,2) + 99.99::DECIMAL(4,2)` -> 199.98
- `(9.9::DECIMAL(2,1) * 9.9::DECIMAL(2,1))::DECIMAL(2,1)` -> ERROR: Conversion Error: Casting value "98.01" to type DECIMAL(2,1) failed: value is out of range

## K. Arrow round-trip (ingestion boundary)

- pyarrow schema: `d: decimal128(10, 2); f: double; i: int64`
- DuckDB sees types: ('DECIMAL(10,2)', 'DOUBLE', 'BIGINT')
- values (d, f, i, d IS NULL, isnan(f)): [(None, 1.5, 1, True, False), (Decimal('1.25'), None, None, False, None), (Decimal('-3.50'), nan, 3, False, True)]
- Arrow validity-null arrives as SQL NULL; Arrow float NaN arrives as
  NaN (a value) — the Seine boundary normalizes NaN -> NULL for
  NULLABLE float fields per D-095 (and keeps rejecting NaN-as-null
  confusion for non-nullable ones).
